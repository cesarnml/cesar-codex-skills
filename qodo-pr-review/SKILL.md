# Qodo PR Review

Use this skill when the user wants the latest `qodo-code-review` PR comments, including shorthand requests like `qcr`.

## Workflow

1. Confirm `gh auth status` succeeds if GitHub access is uncertain.
2. Resolve the PR number with `gh pr view`.
   If the user names a PR, use it directly. Otherwise default to the current branch's PR.
3. Fetch Qodo comments with `gh`, not repo-local helper scripts.
   Use both:
   - `gh pr view <pr> --json comments,reviews,latestReviews,url` for top-level PR comments and review summaries
   - `gh api repos/<owner>/<repo>/pulls/<pr>/comments --paginate` for inline review comments
   Filter to authors whose login starts with `qodo-code-review`.
   Ignore missing helper scripts; do not waste time searching for them.
4. Triage each Qodo comment before recommending any action.
   Classify it as actionable, stale, wrong, over-scoped, or out of scope.
   Ignore the standard autogenerated summary, walkthrough, and file-change overview unless the user explicitly asks for that summary noise.
5. Treat Qodo comments as advisory, not authoritative.
   They are never gospel and should not be implemented blindly.
6. Apply the repo's `ai-cr` policy explicitly.
   Push back on stale, over-scoped, unnecessary, or policy-conflicting suggestions.
7. If the user explicitly approves your Qodo triage and asks you to make the patch, do the patch immediately instead of waiting for another confirmation.
8. After applying an approved Qodo patch:
   - run the smallest relevant verification
   - commit the patch changes in the current repo
   - push the current branch so the PR updates automatically
9. If you update this skill itself, commit and push the skill repo as part of the same request.

## Output expectations

- Start with whether Qodo left any comments.
- Filter out the standard autogenerated summary comment by default and report only actionable review items, plus any stale or wrong items worth pushing back on.
- State which comments are actionable and which should be pushed back on.
- Group by unresolved actionable issue, not by raw API payload.
- Include file and line when available.
- Call out stale or already-addressed comments explicitly instead of blindly implementing them.
- When recommending no change, say why the comment should be rejected.
- When the user has already approved your take, default to implementing the approved actionable items and pushing the branch update rather than asking again.

## Notes

- Default to the current branch's PR if no PR number is given.
- Fetch both top-level PR comments and inline review comments from logins that start with `qodo-code-review`.
- Prefer direct `gh` usage because repo-local helper scripts may not exist in every worktree.
